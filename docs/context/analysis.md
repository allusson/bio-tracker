The analysis/ directory contains three Jupyter notebooks for exploratory data analysis and pipeline validation. All three connect directly to Databricks to query real deployment data. Plots are saved to outputs/ (gitignored). Notebooks must never be committed with cell outputs because query results contain real device and patient identifiers. Clear outputs before every commit using jupyter nbconvert --clear-output --inplace or the Clear All Outputs menu option.

Every notebook requires the same credential setup. load_dotenv("../.env") must be the very first code cell that executes, before any os.getenv() call that feeds into sql.connect(). The .env file lives at the repo root, one directory above analysis/. Running the connection cell before the dotenv cell produces a connection with None credentials and a confusing authentication error. The connector is databricks-sql-connector (from databricks import sql), not databricks-sdk. All three notebooks define a local helper function run_query(query) -> pd.DataFrame that wraps cursor.execute() and converts Row objects to dicts via .asDict() before constructing the DataFrame.

01_data_overview.ipynb queries etl_device_telemetry_bronze.gateway_scan_telemetry and produces a complete characterisation of the real dataset across five sections. The gateway overview section groups by gateway_id and computes total readings, unique devices, mean RSSI, earliest timestamp, and latest timestamp. As of the last run: 19,885 unique gateways and 11.4 million total readings; the top two gateways account for approximately 2.4 million readings between them. The RSSI distribution section fetches all filtered RSSI values and fits a normal distribution; as of the last run the mean was -69.3 dBm, median -70.0 dBm, and standard deviation 15.0 dB. The temporal pattern section uses the most active gateway to visualise the ~10-minute reporting cadence per hour. It also plots per-gateway reporting offsets (seconds past the nearest 10-minute boundary) and reveals that each hub has a fixed clock offset baked in at boot time — this is the empirical motivation for the 20-minute bucketing window. The multi-hub visibility section counts distinct gateways per (device, timestamp) pair; as of the last run, 99.5% of raw observations are single-hub and 0.5% have two or more hubs simultaneously. Multi-hub windows are rare in the raw data but become more frequent after 20-minute bucketing. All queries filter WHERE gateway_id != 'bio_id_test_scan_results' and WHERE rssi <= 0 AND rssi >= -100, and use reported_scan_timestamp rather than timestamp.

02_multi_hub_visibility.ipynb documents the clock offset finding in depth and validates that 20-minute bucketing is correct. It requires floor_timestamp() from utils/helpers.py to be working. Section 1 uses a known sample device (ID t42032c01422c1d01380c1401) seen by 10 gateways over a 4-hour window on 2025-03-04. Each gateway forms a clean, evenly-spaced reporting timeline at its own fixed offset. The within-window timestamp spread reaches up to approximately 9 minutes and 25 seconds between the earliest and latest hub report for what is logically the same scan observation. Section 2 applies floor_timestamp() with both 10-minute and 20-minute window sizes to the same data and counts distinct gateways captured per window. With 10-minute bucketing, the clock offset causes gateways for the same logical observation to fall into different buckets, fragmenting RSSI vectors. With 20-minute bucketing, all gateways land in the same bucket. Section 3 pulls one full day of real data (2025-03-04), runs pipeline.wrangler.wrangle() on it, and measures the hub-count distribution in the resulting RSSI vectors to validate vector completeness in practice.

03_synthetic_vs_real.ipynb is a skeleton. All analysis cells contain only pass stubs. Its purpose is a side-by-side comparison of synthetic generator output against real deployment data to calibrate synthetic_data/config.py. The planned sections are: load a real data sample from Databricks; load synthetic data via generate(); compare RSSI distributions (calibration target: synthetic mean approximately -69 dBm, sigma approximately 15 dB to match real data); compare multi-hub visibility rates; compare temporal density and reporting cadence; compare gateway clock offset behaviour (real hubs have fixed per-hub offsets, synthetic hubs currently report at identical timestamps); and produce calibration recommendations for synthetic_data/config.py based on findings.

Saved outputs follow the naming convention NN_description.png where NN matches the notebook number. Notebook 01 produces: 01_gateway_reading_counts.png (top 15 gateways by reading count), 01_rssi_distribution.png (RSSI histogram with normal fit), 01_temporal_pattern.png (readings per hour for top gateway), 01_gateway_clock_offsets.png (per-gateway reporting offset strip plot), 01_multi_hub_visibility.png (hub count distribution in raw data). Notebook 02 produces: 02_gateway_timelines.png (timeline strip chart for the sample device), 02_bucketing_comparison.png (10-minute vs 20-minute gateway capture counts), 02_vector_completeness.png (hub-count distribution after wrangling).

Dependencies: pandas, numpy, matplotlib, seaborn, scipy (for stats.norm.fit), databricks-sql-connector, python-dotenv.
