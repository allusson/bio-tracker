The synthetic_data/ directory generates a synthetic RSSI dataset that mimics real BioHub scan telemetry. It is used for offline pipeline development when a Databricks connection is unavailable or when a reproducible dataset with known ground-truth device positions is needed.

synthetic_data/config.py holds all tunable simulation constants. SPACE_WIDTH defaults to 100 metres, SPACE_HEIGHT to 60 metres. N_HUBS defaults to 4. HUB_POSITIONS defaults to None, which triggers automatic grid placement; set it to a list of (x, y) tuples to override. N_DEVICES defaults to 6. SIM_DURATION_HOURS defaults to 8. RSSI_REF defaults to -40 dBm, which is the expected signal strength at exactly 1 metre from a hub. PATH_LOSS_EXP defaults to 3.0; 2.0 represents free space, 3.0-4.0 represents indoor environments with walls and furniture. NOISE_STD defaults to 4.0 dB, the standard deviation of Gaussian noise added to each reading. RSSI_MIN_THRESHOLD defaults to -90 dBm; readings weaker than this are not recorded, simulating the receiver sensitivity floor. SCAN_WINDOW_MINUTES defaults to 20, matching the real hub reporting interval.

synthetic_data/generator.py contains the simulation engine. It has two functions and a CLI entry point.

_place_hubs() returns a dict mapping hub ID strings (e.g. "Hub-A") to (x, y) float tuples in metres. If config.HUB_POSITIONS is set, those positions are used directly. Otherwise, the function computes an automatic grid: ncols = ceil(sqrt(N)), nrows = ceil(N / ncols), then places each hub at the centre of its grid cell, filling left-to-right, top-to-bottom.

generate() is the main simulation entry point. It uses a fixed random seed of 42, making all output fully deterministic. It builds a sequence of timestamps starting at 2025-01-01 08:00:00, with one timestamp every SCAN_WINDOW_MINUTES minutes for SIM_DURATION_HOURS hours, producing n_steps = SIM_DURATION_HOURS * 60 / SCAN_WINDOW_MINUTES total steps. Each device starts at a random (x, y) position within the floor bounds. At each timestamp, every device position is updated by a random walk: displacement drawn from N(0, 3.0^2) in both x and y, clamped to the floor boundaries. The step_size of 3.0 m/step is hardcoded inside generate(), not in config.py. For each hub-device pair, Euclidean distance is computed and floored to 0.1 m to avoid log10(0). RSSI is computed using the log-distance path loss model: RSSI(d) = RSSI_REF - 10 * PATH_LOSS_EXP * log10(d). Gaussian noise from N(0, NOISE_STD^2) is then added. Readings below RSSI_MIN_THRESHOLD are discarded. Surviving readings are accumulated in a plain Python list and converted to a DataFrame in a single pd.DataFrame() call at the end — not appended row by row, which would cause O(n^2) memory copies. Output columns are gateway_id, timestamp, device_id, rssi (rounded to 2 decimal places, always <= 0).

main() is the CLI entry point; it calls generate() and saves the result to outputs/synthetic_data.csv.

The log-distance path loss model formula is RSSI(d) = RSSI_REF - 10 * n * log10(d), where RSSI_REF = -40 dBm is the signal at 1 m, n = PATH_LOSS_EXP = 3.0 is the indoor path loss exponent, and d is distance in metres. Real-world RSSI fluctuates due to multipath and body shadowing; NOISE_STD = 4.0 dB Gaussian noise approximates this.

The synthetic generator does not model per-hub clock offsets. Real BioHub nodes have fixed per-hub clock offsets spanning up to ~9.5 minutes; the generator assigns all hubs the same timestamp at each step. Whether per-hub offsets need to be added to the generator for accurate calibration is an open question that analysis/03_synthetic_vs_real.ipynb (currently a skeleton) is intended to answer. The movement step_size of 3.0 m/step is hardcoded inside generate() and not exposed in config.py. The seed=42 makes output deterministic; remove or parameterise it to generate varied datasets. The docs/js/rssi.js sandbox uses different constants (RSSI_AT_1M = -45, PATH_LOSS_EXP = 2.7, wall attenuation = 12 dBm) tuned for visual demo purposes, not calibrated against real data.
