This is an unsupervised indoor BLE localization pipeline for Bio Button wearable devices deployed in a hospital setting. Fixed BioHub gateway nodes passively scan BioButton devices and record Bluetooth RSSI readings. The core enabling property is simultaneous multi-hub visibility: when multiple hubs hear the same device at the same time, each observation becomes an RSSI vector (one value per hub) rather than a scalar reading. These vectors carry spatial information without requiring a floor plan or any labeled ground truth.

The pipeline has four stages. Stage 1 is RSSI-vector clustering (k-means, GMM, or DBSCAN). Stage 2 is HMM zone transition modeling with Viterbi decoding. Stage 3 (planned) is MDS pseudo-map construction from pairwise RSSI dissimilarity. Stage 4 (planned) is an optional particle filter layer.

Data flows in one of two paths. The synthetic path runs data/synthetic_data/generator.py, which produces a long-format DataFrame directly. The real path runs data/real_data/databricks_pull.py (stub) followed by data/real_data/preprocessor.py, which also produces a long-format DataFrame. Both paths output the same schema and feed into src/pipeline/wrangler.py. Wrangler feeds clustering.py, which feeds hmm.py.

Module statuses: data/synthetic_data/generator.py is implemented. data/real_data/preprocessor.py is implemented. src/pipeline/wrangler.py is implemented. src/pipeline/clustering.py is a stub. src/pipeline/hmm.py is a stub. data/real_data/databricks_pull.py is a stub. utils/helpers.py is student implementation in progress. notebooks/01_data_overview.ipynb is implemented. notebooks/02_multi_hub_visibility.ipynb is implemented but requires floor_timestamp() to be working. notebooks/03_synthetic_vs_real.ipynb is a skeleton only. notebooks/04_windowing_reassessment.ipynb is in progress — structured four-thread research notebook reassessing the 20-minute bucketing window following supervisor feedback; Threads 1 (timing characterisation) and 2 (window-width sweep) are the minimum deliverable, Threads 3 (alternative grouping) and 4 (confidence scoring) are stretch goals.

The long-format schema used throughout the pipeline (raw and preprocessed): gateway_id (str, BioHub identifier), timestamp (datetime, UTC-aware after preprocessing), device_id (str, BioButton identifier), rssi (float, signal strength in dBm, always <= 0).

The wide-format schema produced by the wrangler: a MultiIndex of (device_id, time_window) as rows, with hub IDs as columns. NaN where a hub did not see the device in that window.

The central config.py at the repo root controls: USE_SYNTHETIC (bool, toggles data source), WINDOW_MINUTES (default 20, size of time buckets), WINDOW_TOLERANCE_MINUTES (default 2, not yet used in wrangler), MIN_HUBS_PER_WINDOW (default 2, minimum hubs required to keep a window), CLUSTERING_METHOD (default "kmeans", options: "kmeans", "gmm", "dbscan"), N_ZONES (default 4, used by k-means and GMM), HMM_N_ITER (default 100), OUTPUT_DIR (default "outputs/").

The .env file must use plain KEY=VALUE format with no quotes and no spaces around the equals sign. The three required keys are DATABRICKS_SERVER_HOSTNAME, DATABRICKS_HTTP_PATH, and DATABRICKS_TOKEN. The connector used throughout is databricks-sql-connector (from databricks import sql), not databricks-sdk. requirements.txt currently lists databricks-sdk, which is incorrect — databricks-sql-connector must be installed separately.

Every notebook must call load_dotenv("../.env") in its first code cell before any os.getenv() call that feeds into sql.connect(). Running cells out of order produces a connection with None credentials.

Critical gotchas. aggfunc="last" in wrangler.py means last by DataFrame row order, not last chronologically — upstream sort by (device_id, timestamp) must be verified before trusting deduplication on real data. The correct time column in etl_device_telemetry_bronze.gateway_scan_telemetry is reported_scan_timestamp; the timestamp column is a pipeline ingestion artifact, not the scan time. The gateway bio_id_test_scan_results is a test fixture with artificial RSSI readings (~-20 dBm) and must always be excluded from queries. The 20-minute bucketing window is required because BioHub nodes have independent fixed clock offsets spanning up to ~9.5 minutes across the hub population, meaning a 10-minute window splits hub reports for the same logical scan cycle across two different buckets.
