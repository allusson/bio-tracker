root

This is an unsupervised indoor BLE localization pipeline for Bio Button wearable devices deployed in a hospital setting. Fixed BioHub gateway nodes passively scan BioButton devices and record Bluetooth RSSI readings. The core enabling property is simultaneous multi-hub visibility: when multiple hubs hear the same device at the same time, each observation becomes an RSSI vector (one value per hub) rather than a scalar reading. These vectors carry spatial information without requiring a floor plan or any labeled ground truth.

The pipeline has four stages. Stage 1 is RSSI-vector clustering (k-means, GMM, or DBSCAN). Stage 2 is HMM zone transition modeling with Viterbi decoding. Stage 3 (planned) is MDS pseudo-map construction from pairwise RSSI dissimilarity. Stage 4 (planned) is an optional particle filter layer.

Data flows in one of two paths. The synthetic path runs synthetic_data/generator.py, which produces a long-format DataFrame directly. The real path runs real_data/databricks_pull.py (stub) followed by real_data/preprocessor.py, which also produces a long-format DataFrame. Both paths output the same schema and feed into pipeline/wrangler.py. Wrangler feeds clustering.py, which feeds hmm.py.

Module statuses: synthetic_data/generator.py is implemented. real_data/preprocessor.py is implemented. pipeline/wrangler.py is implemented. pipeline/clustering.py is a stub. pipeline/hmm.py is a stub. real_data/databricks_pull.py is a stub. utils/helpers.py is student implementation in progress. analysis/01_data_overview.ipynb is implemented. analysis/02_multi_hub_visibility.ipynb is implemented but requires floor_timestamp() to be working. analysis/03_synthetic_vs_real.ipynb is a skeleton only.

The long-format schema used throughout the pipeline (raw and preprocessed): gateway_id (str, BioHub identifier), timestamp (datetime, UTC-aware after preprocessing), device_id (str, BioButton identifier), rssi (float, signal strength in dBm, always <= 0).

The wide-format schema produced by the wrangler: a MultiIndex of (device_id, time_window) as rows, with hub IDs as columns. NaN where a hub did not see the device in that window.

The central config.py at the repo root controls: USE_SYNTHETIC (bool, toggles data source), WINDOW_MINUTES (default 20, size of time buckets), WINDOW_TOLERANCE_MINUTES (default 2, not yet used in wrangler), MIN_HUBS_PER_WINDOW (default 2, minimum hubs required to keep a window), CLUSTERING_METHOD (default "kmeans", options: "kmeans", "gmm", "dbscan"), N_ZONES (default 4, used by k-means and GMM), HMM_N_ITER (default 100), OUTPUT_DIR (default "outputs/").

The .env file must use plain KEY=VALUE format with no quotes and no spaces around the equals sign. The three required keys are DATABRICKS_SERVER_HOSTNAME, DATABRICKS_HTTP_PATH, and DATABRICKS_TOKEN. The connector used throughout is databricks-sql-connector (from databricks import sql), not databricks-sdk. requirements.txt currently lists databricks-sdk, which is incorrect — databricks-sql-connector must be installed separately.

Every notebook must call load_dotenv("../.env") in its first code cell before any os.getenv() call that feeds into sql.connect(). Running cells out of order produces a connection with None credentials.

Critical gotchas. aggfunc="last" in wrangler.py means last by DataFrame row order, not last chronologically — upstream sort by (device_id, timestamp) must be verified before trusting deduplication on real data. The correct time column in etl_device_telemetry_bronze.gateway_scan_telemetry is reported_scan_timestamp; the timestamp column is a pipeline ingestion artifact, not the scan time. The gateway bio_id_test_scan_results is a test fixture with artificial RSSI readings (~-20 dBm) and must always be excluded from queries. The 20-minute bucketing window is required because BioHub nodes have independent fixed clock offsets spanning up to ~9.5 minutes across the hub population, meaning a 10-minute window splits hub reports for the same logical scan cycle across two different buckets.


synthetic_data

The synthetic_data/ directory generates a synthetic RSSI dataset that mimics real BioHub scan telemetry. It is used for offline pipeline development when a Databricks connection is unavailable or when a reproducible dataset with known ground-truth device positions is needed.

synthetic_data/config.py holds all tunable simulation constants. SPACE_WIDTH defaults to 100 metres, SPACE_HEIGHT to 60 metres. N_HUBS defaults to 4. HUB_POSITIONS defaults to None, which triggers automatic grid placement; set it to a list of (x, y) tuples to override. N_DEVICES defaults to 6. SIM_DURATION_HOURS defaults to 8. RSSI_REF defaults to -40 dBm, which is the expected signal strength at exactly 1 metre from a hub. PATH_LOSS_EXP defaults to 3.0; 2.0 represents free space, 3.0-4.0 represents indoor environments with walls and furniture. NOISE_STD defaults to 4.0 dB, the standard deviation of Gaussian noise added to each reading. RSSI_MIN_THRESHOLD defaults to -90 dBm; readings weaker than this are not recorded, simulating the receiver sensitivity floor. SCAN_WINDOW_MINUTES defaults to 20, matching the real hub reporting interval.

synthetic_data/generator.py contains the simulation engine. It has two functions and a CLI entry point.

_place_hubs() returns a dict mapping hub ID strings (e.g. "Hub-A") to (x, y) float tuples in metres. If config.HUB_POSITIONS is set, those positions are used directly. Otherwise, the function computes an automatic grid: ncols = ceil(sqrt(N)), nrows = ceil(N / ncols), then places each hub at the centre of its grid cell, filling left-to-right, top-to-bottom.

generate() is the main simulation entry point. It uses a fixed random seed of 42, making all output fully deterministic. It builds a sequence of timestamps starting at 2025-01-01 08:00:00, with one timestamp every SCAN_WINDOW_MINUTES minutes for SIM_DURATION_HOURS hours, producing n_steps = SIM_DURATION_HOURS * 60 / SCAN_WINDOW_MINUTES total steps. Each device starts at a random (x, y) position within the floor bounds. At each timestamp, every device position is updated by a random walk: displacement drawn from N(0, 3.0^2) in both x and y, clamped to the floor boundaries. The step_size of 3.0 m/step is hardcoded inside generate(), not in config.py. For each hub-device pair, Euclidean distance is computed and floored to 0.1 m to avoid log10(0). RSSI is computed using the log-distance path loss model: RSSI(d) = RSSI_REF - 10 * PATH_LOSS_EXP * log10(d). Gaussian noise from N(0, NOISE_STD^2) is then added. Readings below RSSI_MIN_THRESHOLD are discarded. Surviving readings are accumulated in a plain Python list and converted to a DataFrame in a single pd.DataFrame() call at the end — not appended row by row, which would cause O(n^2) memory copies. Output columns are gateway_id, timestamp, device_id, rssi (rounded to 2 decimal places, always <= 0).

main() is the CLI entry point; it calls generate() and saves the result to outputs/synthetic_data.csv.

The log-distance path loss model formula is RSSI(d) = RSSI_REF - 10 * n * log10(d), where RSSI_REF = -40 dBm is the signal at 1 m, n = PATH_LOSS_EXP = 3.0 is the indoor path loss exponent, and d is distance in metres. Real-world RSSI fluctuates due to multipath and body shadowing; NOISE_STD = 4.0 dB Gaussian noise approximates this.

The synthetic generator does not model per-hub clock offsets. Real BioHub nodes have fixed per-hub clock offsets spanning up to ~9.5 minutes; the generator assigns all hubs the same timestamp at each step. Whether per-hub offsets need to be added to the generator for accurate calibration is an open question that analysis/03_synthetic_vs_real.ipynb (currently a skeleton) is intended to answer. The movement step_size of 3.0 m/step is hardcoded inside generate() and not exposed in config.py. The seed=42 makes output deterministic; remove or parameterise it to generate varied datasets. The docs/js/rssi.js sandbox uses different constants (RSSI_AT_1M = -45, PATH_LOSS_EXP = 2.7, wall attenuation = 12 dBm) tuned for visual demo purposes, not calibrated against real data.


real_data

The real_data/ directory handles Databricks connectivity and conversion of raw scan telemetry into the standard pipeline schema. Its two modules are the real-data counterpart to synthetic_data/generator.py: together they produce the same long-format gateway_id, timestamp, device_id, rssi DataFrame that the rest of the pipeline consumes.

real_data/databricks_pull.py is a stub not yet implemented. When implemented, it will connect to Databricks using credentials from environment variables, query etl_device_telemetry_bronze.gateway_scan_telemetry, and return a raw DataFrame containing gateway_id, reported_scan_timestamp, scanned_device_id, and rssi. The planned function signature is pull(start_date=None, end_date=None) -> pd.DataFrame, accepting optional ISO date strings to restrict the query to a time range. Every query must include WHERE gateway_id != 'bio_id_test_scan_results' to exclude the test fixture gateway, which produces artificial RSSI readings around -20 dBm that will corrupt any analysis. The connector is databricks-sql-connector (from databricks import sql), not databricks-sdk. requirements.txt currently lists databricks-sdk, which is incorrect.

real_data/preprocessor.py is implemented. It exports a single function preprocess(df) that accepts the raw Databricks DataFrame and returns a clean DataFrame ready for pipeline/wrangler.py.

preprocess() performs seven steps in order. Step 1: select only the four relevant columns (gateway_id, reported_scan_timestamp, scanned_device_id, rssi), dropping everything else to save memory. Step 2: rename reported_scan_timestamp to timestamp and scanned_device_id to device_id so the rest of the pipeline uses consistent internal names. Step 3: parse the timestamp column to UTC-aware pd.Timestamp objects using pd.to_datetime(..., utc=True). Without the utc=True argument, tz-naive strings arriving from Databricks would break comparisons against tz-aware timestamps later in the pipeline. Step 4: drop rows where gateway_id, device_id, or rssi is null. Step 5: drop rows with physically implausible RSSI — readings above 0 dBm are physically impossible for BLE, and readings below -100 dBm are below the sensitivity floor of every known BLE chipset; valid range is [-100, 0]. Step 6: sort by (device_id, timestamp) so each device's scan history is in chronological order. This sort is required for aggfunc="last" in wrangler.py to correctly represent the most-recent reading in a window — if anything reshuffles rows between preprocessing and wrangling, the "last" reading will not be the chronologically most recent one. Step 7: print a summary of row count, unique devices, unique gateways, and date range.

The output schema after preprocessing: gateway_id (str, BioHub identifier), timestamp (pd.Timestamp, UTC-aware), device_id (str, BioButton identifier), rssi (float, dBm, guaranteed in [-100, 0]).

Critical gotchas. The correct timestamp column in gateway_scan_telemetry is reported_scan_timestamp. The timestamp column in that same table is a pipeline ingestion artifact recording when the row landed in the lakehouse, not when the hub actually scanned the device. Using timestamp instead of reported_scan_timestamp will produce incorrect time windows. The gateway bio_id_test_scan_results must always be excluded — its readings are synthetic fixtures around -20 dBm and will distort every downstream metric. The sort in preprocess() is load-bearing for wrangler.py correctness: aggfunc="last" in pd.pivot_table() is row-order dependent, not timestamp-order dependent.


pipeline

The pipeline/ directory contains the core algorithm stages of the localization pipeline. Each module takes the output of the previous stage and adds one layer of inference. All stages are controlled by config.py at the repo root.

pipeline/wrangler.py implements Stage 1: converting the raw long-format scan table into the wide-format RSSI vector table that the clustering stage needs. It exports a single function wrangle(df) that accepts a long-format DataFrame (gateway_id, timestamp, device_id, rssi) and returns a pivoted DataFrame indexed by (device_id, time_window) with hub IDs as columns.

wrangle() performs four steps. First, it floors each timestamp to the nearest WINDOW_MINUTES boundary by calling floor_timestamp() from utils/helpers.py and storing the result in a new time_window column. Second, it pivots the long table to wide format using pd.pivot_table() with index=(device_id, time_window), columns=gateway_id, values=rssi, and aggfunc="last". NaN appears wherever a hub did not see the device in that window. Third, it filters out rows (device-window pairs) where the number of non-NaN hub values is below MIN_HUBS_PER_WINDOW; a vector with only one hub reading lacks directional information for localization. Fourth, it prints a diagnostic summary: total windows before and after filtering, percentage with two or more hubs, and an ASCII bar chart of hub-count distribution.

The aggfunc="last" behavior is critical and non-obvious: pd.pivot_table() keeps the last row in current DataFrame order, not the row with the latest timestamp. It does not sort internally. The preprocessor sorts by (device_id, timestamp) before data reaches wrangle(), so ordering is correct for preprocessed real data. If an unsorted DataFrame is passed directly, the "last" reading may not be chronologically most recent.

The 20-minute window is required for real data and is not configurable downward. BioHub nodes have independent fixed clock offsets reaching up to ~9.5 minutes across the hub population. This is a reporting artifact — hub report timestamps cannot be interpreted as the moment a device was exclusively in one hub's range. A 10-minute window splits hub reports for the same logical scan cycle across different buckets, fragmenting RSSI vectors. The empirical evidence is in analysis/02_multi_hub_visibility.ipynb.

WINDOW_TOLERANCE_MINUTES (default 2) is defined in config.py but is not yet used inside wrangler.py.

The output is a MultiIndex DataFrame. Rows are indexed by (device_id, time_window). Columns are hub IDs. Values are RSSI in dBm. NaN means that hub did not see the device in that window. Only rows with at least MIN_HUBS_PER_WINDOW non-NaN values are included.

pipeline/clustering.py is a stub. When implemented, it will accept the wide-format RSSI vector DataFrame and return it with an added zone column. NaN values must be imputed or masked before vectors are passed to scikit-learn estimators, which do not accept NaN inputs. Three methods are planned, controlled by config.CLUSTERING_METHOD: kmeans (fast baseline, hard zone assignments, requires N_ZONES), gmm (soft probabilistic assignments, better upstream input for the HMM, also requires N_ZONES), and dbscan (density-based, automatically determines number of zones, marks noise observations as zone -1).

pipeline/hmm.py is a stub. When implemented, it will accept the DataFrame with the zone column from clustering.py and return it with an added smoothed_zone column. Hidden states are the zones discovered by clustering. The transition matrix will be estimated from empirical zone-to-zone transition frequencies in the data. Emissions will be modeled from RSSI vectors or GMM cluster probabilities. Viterbi decoding via hmmlearn will recover the most probable zone sequence for each device over time.

Configuration parameters from config.py relevant to this directory: WINDOW_MINUTES (default 20) controls bucket size in wrangler. MIN_HUBS_PER_WINDOW (default 2) controls the filter threshold in wrangler. CLUSTERING_METHOD (default "kmeans") controls which algorithm clustering.py will use. N_ZONES (default 4) sets the number of clusters for k-means and GMM. HMM_N_ITER (default 100) sets training iterations for the HMM.

Dependencies: pandas for pivoting, filtering, and MultiIndex handling in wrangler.py; scikit-learn for k-means and GMM in clustering.py (planned); hmmlearn for HMM and Viterbi in hmm.py (planned).


utils

The utils/ directory contains shared utility functions used across pipeline/, real_data/, and analysis/. All functions live in utils/helpers.py. All three functions are student implementations in progress, written as a Python learning exercise. Each has a complete signature, docstring, and pseudocode in the source, but the implementations should not be assumed complete or stable.

floor_timestamp(ts: pd.Timestamp, window_minutes: int) -> pd.Timestamp floors a timestamp down to the nearest window_minutes boundary. Given ts=08:23:45 and window_minutes=20, the correct result is 08:20:00. The intended algorithm is: compute remainder = ts.minute % window_minutes to find how many minutes past the last boundary the timestamp is, subtract timedelta(minutes=remainder) to land on the boundary, then zero out seconds and microseconds so the result is a clean tick. This function works correctly for any window_minutes value that evenly divides 60 (1, 2, 3, 4, 5, 6, 10, 12, 15, 20, 30, 60). For values that do not divide 60 evenly, it still produces a repeatable bucket but buckets will not align to clock hours. This is the function called by pipeline/wrangler.py to assign every raw observation to its time window, and also used directly by analysis/02_multi_hub_visibility.ipynb for the bucketing comparison. Notebook 02 will fail unless this function is working.

rssi_to_distance(rssi: float, rssi_ref: float, path_loss_exp: float) -> float converts an RSSI reading to an estimated distance in metres using the inverse log-distance path loss model. The forward model is RSSI(d) = RSSI_ref - 10 * n * log10(d). Rearranged to solve for d: d = 10 ^ ((RSSI_ref - RSSI) / (10 * n)). The argument rssi is the measured signal strength in dBm, always negative (e.g. -65.0). rssi_ref is the expected RSSI at exactly 1 metre from the transmitter (e.g. -40.0 dBm). path_loss_exp is the path loss exponent n; 2.0 for free space, 3.0-4.0 for indoor environments. This function will be useful for Stage 3 (MDS pseudo-map construction), where hub-device RSSI readings need to be converted to pairwise distance estimates for multidimensional scaling.

load_data(use_synthetic: bool | None = None) -> pd.DataFrame is a unified data loader that returns a clean long-format DataFrame regardless of source. If use_synthetic=True, it calls synthetic_data.generator.generate(). If use_synthetic=False, it calls real_data.databricks_pull.pull() followed by real_data.preprocessor.preprocess(). If use_synthetic=None (the default), it reads config.USE_SYNTHETIC to determine which branch to take. Both branches must return a DataFrame with identical column names (gateway_id, timestamp, device_id, rssi) so the rest of the pipeline is data-source agnostic.

Dependencies: pandas for the pd.Timestamp type used in floor_timestamp; datetime (standard library) for timedelta used in floor_timestamp.

The analysis/ directory contains three Jupyter notebooks for exploratory data analysis and pipeline validation. All three connect directly to Databricks to query real deployment data. Plots are saved to outputs/ (gitignored). Notebooks must never be committed with cell outputs because query results contain real device and patient identifiers. Clear outputs before every commit using jupyter nbconvert --clear-output --inplace or the Clear All Outputs menu option.

Every notebook requires the same credential setup. load_dotenv("../.env") must be the very first code cell that executes, before any os.getenv() call that feeds into sql.connect(). The .env file lives at the repo root, one directory above analysis/. Running the connection cell before the dotenv cell produces a connection with None credentials and a confusing authentication error. The connector is databricks-sql-connector (from databricks import sql), not databricks-sdk. All three notebooks define a local helper function run_query(query) -> pd.DataFrame that wraps cursor.execute() and converts Row objects to dicts via .asDict() before constructing the DataFrame.

01_data_overview.ipynb queries etl_device_telemetry_bronze.gateway_scan_telemetry and produces a complete characterisation of the real dataset across five sections. The gateway overview section groups by gateway_id and computes total readings, unique devices, mean RSSI, earliest timestamp, and latest timestamp. As of the last run: 19,885 unique gateways and 11.4 million total readings; the top two gateways account for approximately 2.4 million readings between them. The RSSI distribution section fetches all filtered RSSI values and fits a normal distribution; as of the last run the mean was -69.3 dBm, median -70.0 dBm, and standard deviation 15.0 dB. The temporal pattern section uses the most active gateway to visualise the ~10-minute reporting cadence per hour. It also plots per-gateway reporting offsets (seconds past the nearest 10-minute boundary) and reveals that each hub has a fixed clock offset baked in at boot time — this is the empirical motivation for the 20-minute bucketing window. The multi-hub visibility section counts distinct gateways per (device, timestamp) pair; as of the last run, 99.5% of raw observations are single-hub and 0.5% have two or more hubs simultaneously. Multi-hub windows are rare in the raw data but become more frequent after 20-minute bucketing. All queries filter WHERE gateway_id != 'bio_id_test_scan_results' and WHERE rssi <= 0 AND rssi >= -100, and use reported_scan_timestamp rather than timestamp.

02_multi_hub_visibility.ipynb documents the clock offset finding in depth and validates that 20-minute bucketing is correct. It requires floor_timestamp() from utils/helpers.py to be working. Section 1 uses a known sample device (ID t42032c01422c1d01380c1401) seen by 10 gateways over a 4-hour window on 2025-03-04. Each gateway forms a clean, evenly-spaced reporting timeline at its own fixed offset. The within-window timestamp spread reaches up to approximately 9 minutes and 25 seconds between the earliest and latest hub report for what is logically the same scan observation. Section 2 applies floor_timestamp() with both 10-minute and 20-minute window sizes to the same data and counts distinct gateways captured per window. With 10-minute bucketing, the clock offset causes gateways for the same logical observation to fall into different buckets, fragmenting RSSI vectors. With 20-minute bucketing, all gateways land in the same bucket. Section 3 pulls one full day of real data (2025-03-04), runs pipeline.wrangler.wrangle() on it, and measures the hub-count distribution in the resulting RSSI vectors to validate vector completeness in practice.

03_synthetic_vs_real.ipynb is a skeleton. All analysis cells contain only pass stubs. Its purpose is a side-by-side comparison of synthetic generator output against real deployment data to calibrate synthetic_data/config.py. The planned sections are: load a real data sample from Databricks; load synthetic data via generate(); compare RSSI distributions (calibration target: synthetic mean approximately -69 dBm, sigma approximately 15 dB to match real data); compare multi-hub visibility rates; compare temporal density and reporting cadence; compare gateway clock offset behaviour (real hubs have fixed per-hub offsets, synthetic hubs currently report at identical timestamps); and produce calibration recommendations for synthetic_data/config.py based on findings.

Notebook 01 produces: 01_gateway_reading_counts.png (top 15 gateways by reading count), 01_rssi_distribution.png (RSSI histogram with normal fit), 01_temporal_pattern.png (readings per hour for top gateway), 01_gateway_clock_offsets.png (per-gateway reporting offset strip plot), 01_multi_hub_visibility.png (hub count distribution in raw data). demo

The docs/ directory contains a standalone browser-based BLE Coverage Sandbox. It is not part of the Python pipeline. It is a pure-JavaScript tool that runs directly in any modern browser by opening docs/index.html — no build step, no server, no dependencies. It is useful for building intuition about how hub placement affects RSSI coverage overlap and zone separation, which are the same properties that determine RSSI vector quality in the Python pipeline.

index.html is the entry point. It defines the toolbar with mode buttons (Add Hub, Add Device, Add Wall, Select/Move, Delete), a radius slider controlling the detection radius (40-420 px range, default 150 px), a signal rings toggle, and zone compute/clear buttons. It also defines the canvas element, a tooltip overlay showing hub name and position, and keyboard shortcut hints. Scripts are loaded in dependency order: state.js, then rssi.js, then zones.js, then canvas.js, then tools.js, then ui.js. Keyboard shortcuts are H (add hub), D (add device), W (add wall), S (select/move), X (delete), Esc (cancel or deselect), Delete (remove selected), and right-click (cancel wall in progress).

styles.css provides visual styling for the toolbar, canvas, and tooltip.

state.js holds the global application state object STATE. It contains STATE.hubs (array of hub objects with x, y, and label), STATE.devices (array of device objects), STATE.walls (array of wall segments with x1, y1, x2, y2), STATE.radius (current detection radius in pixels), STATE.zonesComputed (boolean), and STATE.zoneGrid (array of grid point assignments). All other modules read and write this shared object.

rssi.js contains pure signal strength math with no DOM interaction. Constants: PIXELS_PER_METER = 20 (scale factor, so 20 px equals 1 metre), PATH_LOSS_EXP = 2.7 (log-distance path loss exponent), RSSI_AT_1M = -45 dBm (reference RSSI at 1 metre), WALL_ATTENUATION_DB = 12 dBm (signal penalty per intersecting wall segment). The computeRSSI(dev, hub) function returns an object with three fields: freeSpace (RSSI computed purely from distance using the log-distance model: RSSI_AT_1M - 10 * PATH_LOSS_EXP * log10(distance_metres)), attenuated (freeSpace minus wallCount * WALL_ATTENUATION_DB for each wall that crosses the dev-hub line), and wallCount (number of wall segments intersected). The segmentsIntersect(ax, ay, bx, by, cx, cy, dx, dy) function performs a parametric segment-segment intersection test using cross products. It returns the parameter t along segment AB at the intersection point if the segments cross within their extents, or null otherwise. This is used to count how many wall segments cross each dev-hub line. The connectionStrength(dev, hub) function returns a [0, 1] opacity value based on pixel distance relative to STATE.radius, or null if the device is beyond the radius. This is used for rendering signal ring opacity.

zones.js implements zone classification. It samples the canvas on an 8-pixel grid and assigns each grid point to the hub with the highest attenuated RSSI (strongest-hub-wins rule). It uses a 10-colour palette at 20% fill opacity. computeZones() runs the full grid pass, stores the result in STATE.zoneGrid, and pre-renders an offscreen canvas. renderZones() blits the pre-rendered offscreen canvas onto the main canvas; it is called from the main render loop in canvas.js. invalidateZones() clears the cached zone image whenever a hub moves or is added or removed, so no stale zone colours are shown until the user recomputes. clearZones() is the explicit user-initiated clear action.

canvas.js contains the main render loop using requestAnimationFrame. It draws hubs as circles with labels, devices as squares with RSSI readouts, walls as line segments, signal rings, and the zone overlay from zones.js.

tools.js contains mouse event handlers for each editing mode. Mode switches are triggered by toolbar button clicks or keyboard shortcuts.

ui.js handles toolbar wiring: updating button active states on mode switch, syncing the radius slider to STATE.radius, controlling zone button visibility based on STATE.zonesComputed, and dispatching keyboard shortcut events.

The sandbox uses different path loss constants than the Python pipeline. The sandbox uses RSSI_AT_1M = -45 dBm, PATH_LOSS_EXP = 2.7, and wall attenuation of 12 dBm per wall. synthetic_data/config.py uses RSSI_REF = -40 dBm, PATH_LOSS_EXP = 3.0, and no wall attenuation. The sandbox constants are tuned for a visually reasonable interactive demo; the Python generator constants are a separate design choice that should be calibrated against real data via analysis/03_synthetic_vs_real.ipynb.
